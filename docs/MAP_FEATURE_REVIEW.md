# 지도 기능 검토

## 요청 요약

매장리스트 화면에서 현재 지역의 조사 대상 매장을 지도에 표시한다.

- 미완료 마트: 눈에 잘 띄는 색상 마커
- 완료 마트: 회색 계열 마커
- 사용자 위치 허용 시 내 위치 표시
- 내 주변 미완료 매장을 지도에서 쉽게 확인
- 마커 선택 시 매장명, 주소, 완료 여부, 입력 화면 이동 제공

담당자가 실제로 방문할 매장을 먼저 체크할 수 있어야 한다. 담당매장 관리는 메인 지역 카드 메뉴에서 먼저 수행하고, 이후 매장리스트와 매장지도는 담당매장으로 선택된 항목만 표시한다. 기본값은 전체 미선택이며, 지역 통계는 전체 매장 기준을 유지한다.

## 현재 앱 데이터 상태

현재 `SurveyStore`에는 `storeAddress`와 위도/경도 캐시 필드가 있다.

따라서 지도에 마커를 찍으려면 주소를 좌표로 변환하는 지오코딩 단계가 필요하다.
이 좌표는 한 번 변환한 뒤 IndexedDB에 저장해 재사용하는 방식이 적합하다.

권장 데이터 추가:

```ts
type SurveyStore = {
  latitude?: number;
  longitude?: number;
  geocodedAt?: string;
  geocodeStatus?: "성공" | "실패" | "미시도";
};
```

## 구현 가능 여부

가능하다.

다만 GitHub Pages에서 배포하는 공개 프론트엔드 앱이므로 API 키를 숨길 수 없다.
지도/주소 API 키를 사용할 경우 도메인 제한, 사용량 제한, 비용 정책을 반드시 고려해야 한다.

## 구현 방식 후보

### 1안. 카카오맵 또는 네이버지도 API 사용

장점:
- 한국 주소 지오코딩 정확도가 좋다.
- 국내 사용자가 익숙한 지도 화면을 제공할 수 있다.
- 주소 기반 마커 표시와 현재 위치 표시가 자연스럽다.

주의:
- API 키가 필요하다.
- GitHub Pages에 올라가는 키는 완전히 숨길 수 없으므로 도메인 제한이 필요하다.
- 오프라인에서는 지도 타일과 신규 지오코딩이 동작하지 않는다.

### 2안. Leaflet + OpenStreetMap 지도

장점:
- 구현이 단순하다.
- 마커 색상/팝업/내 위치 표시를 자유롭게 만들 수 있다.
- 앱 화면 안에 지도를 쉽게 넣을 수 있다.

주의:
- 주소를 좌표로 바꾸는 지오코딩은 별도 API가 필요하다.
- OpenStreetMap 기본 타일 서버는 대량/상업적 사용에 주의가 필요하다.
- 국내 주소 검색 정확도는 카카오/네이버보다 낮을 수 있다.

### 3안. 좌표 업로드 또는 수동 좌표 캐시

장점:
- 지도 API 의존도를 줄일 수 있다.
- 한 번 만든 좌표를 백업/복원으로 다른 기기에 옮길 수 있다.
- 현장에서는 지오코딩 없이 지도 표시만 하면 되므로 빠르다.

주의:
- 최초 좌표 생성 단계가 필요하다.
- 주소가 변경되면 좌표를 다시 갱신해야 한다.

## 추천 방향

1차 구현은 다음 방식이 가장 현실적이다.

1. 매장 데이터에 위도/경도 캐시 필드 추가
2. 메인 지역 카드 메뉴에 `담당매장 관리` 추가
3. 매장 작업 화면에는 `매장 리스트`, `매장 지도` 보기 전환만 유지
4. 좌표가 없는 매장은 온라인 상태에서 지오코딩 후 IndexedDB에 저장
5. 좌표 변환 실패 매장은 지도 하단에 목록으로 따로 표시
6. 사용자 위치 허용 시 내 위치 마커 표시
7. 완료/미완료 상태별 마커 색상 구분

## 비용/사용량 제한을 줄이는 방향

지도에서 마커가 뜨고 사라지는 기능 자체는 API 비용 없이 구현할 수 있다.
문제는 주소를 좌표로 바꾸는 지오코딩이다.

가장 비용 부담이 적고 안정적인 방향은 다음과 같다.

1. 앱 내부 데이터에 위도/경도 필드를 둔다.
2. 조사표 또는 별도 좌표 파일에 위도/경도 컬럼이 있으면 그대로 가져온다.
3. 위도/경도가 있는 마트만 지도에 즉시 표시한다.
4. 위도/경도가 없는 마트는 지도 아래에 `좌표 없음` 목록으로 표시한다.
5. 좌표 보정은 사무실에서 한 번만 수행하고 IndexedDB/백업에 저장한다.

이렇게 하면 현장에서는 주소검색 API를 호출하지 않고, 이미 저장된 좌표만 지도에 찍는다.

### 무제한에 가장 가까운 1차안

- 지도 라이브러리: Leaflet
- 마커 데이터: 앱 IndexedDB에 저장된 위도/경도
- 마커 상태:
  - 완료: 회색
  - 미완료: 빨강 또는 주황
  - 선택한 마트: 파랑
- 내 위치: 브라우저 위치 권한을 허용한 경우에만 표시
- 주소 지오코딩: 앱 필수 기능에서 제외하고 좌표 컬럼/좌표 백업을 우선 사용

이 방식은 마커 표시 자체에 지도 API 키가 필요하지 않다.
다만 배경 지도 타일은 별도 타일 서버를 사용한다.

### 배경 지도 타일 주의

OpenStreetMap 기본 타일은 API 키가 필요 없지만 완전한 무제한 서비스는 아니다.
공식 정책상 과도한 사용, 대량 다운로드, 오프라인 사전 저장은 제한된다.
따라서 GitHub Pages에서 소규모 현장조사용으로 쓰는 것은 가능성이 높지만, 대량 배포/대량 접속을 전제로 하면 별도 타일 제공자 또는 자체 타일이 필요하다.

### 완전한 외부 API 회피안

외부 주소검색 API와 지도 API 의존을 거의 없애려면 다음 구조가 필요하다.

- 위도/경도 컬럼을 조사표 또는 보조 좌표 파일에 포함
- 앱은 좌표를 읽어서 마커만 표시
- 지도 배경은 Leaflet + 공개 타일 또는 자체 호스팅 타일 사용
- 좌표가 없는 마트는 지도 표시 대상에서 제외하고 목록으로 안내

이 경우 주소를 좌표로 바꾸는 비용과 사용량 문제는 앱 밖의 준비 과정으로 빠진다.

### 앱 안에서 자동 좌표 변환을 넣는 경우

앱에서 주소를 자동 변환하려면 카카오, 네이버, 구글, Nominatim 같은 지오코딩 서비스가 필요하다.
이 방식은 편하지만 API 키, 사용량 제한, 정책 문제가 생긴다.
무료 공용 지오코딩은 대량 변환에 적합하지 않으므로 조사 물량이 많은 경우 권장하지 않는다.

마커 색상 제안:

- 미완료: 빨강 또는 진한 주황
- 완료: 회색
- 현재 선택/포커스 매장: 파랑
- 내 위치: 파랑 원형 마커

## 완료 기준

매장 완료 기준은 기존 앱 기준을 따른다.

- 매장 전경사진 등록
- 해당 매장 물품 입력 완료

이 기준을 만족하면 지도에서 완료 마커로 표시한다.

## 화면 흐름 제안

매장리스트 화면:

- 메인 지역 카드의 `담당매장 관리`에서 지도/리스트에 표시할 매장을 체크한다.
- 매장 작업 화면에서는 상단 탭에서 `매장 리스트`, `매장 지도`를 전환한다.
- 매장 카드 메뉴에서도 `담당매장 포함/제외`를 설정할 수 있다.

지도 화면:

- 상단: 담당매장 수, 완료/미완료 수, 내 위치
- 본문: 지도
- 마커 탭: 매장명, 주소, 완료 여부, `입력` 버튼
- `없는 위치만 가져오기`는 담당매장 중 좌표가 없는 매장만 검색한다.
- `전체 위치 다시 가져오기`는 담당매장 전체 좌표를 다시 검색한다.
- 좌표 검색 시 주소 앞 우편번호와 매대/코너 안내 문구를 제거한다.
- 내 위치가 확인되면 매장지도와 거리순 정렬을 사용할 수 있다. 단, 매장 좌표가 없으면 거리순 정렬은 정확하지 않을 수 있다.
- 무료 공용 Nominatim 정책을 고려해 1초에 1건씩만 검색하고, 성공한 좌표는 IndexedDB에 저장한다.

## 1차 구현 범위

- Leaflet을 사용해 앱 내부 지도 화면을 추가한다.
- Carto 무료 타일을 우선 배경지도로 사용하고, 실패 시 OpenStreetMap 타일로 fallback한다.
- `mapIncluded === true`인 매장을 담당매장으로 본다.
- 엑셀에 위도/경도 컬럼이 있으면 좌표를 읽어 매장에 저장한다.
- 좌표가 있는 매장만 지도에 원형 마커로 표시한다.
- 완료 매장은 회색, 미완료 매장은 빨강, 선택 매장은 파랑으로 표시한다.
- 내 위치 버튼은 브라우저 Geolocation API를 사용한다.
- 좌표 없는 매장은 사용자가 `추가 위치` 또는 `전체 갱신`을 눌렀을 때만 주소 검색을 시도한다.

## 1차 구현 제외

- 자동 일괄 지오코딩
- 지도 타일 오프라인 저장
- 외부 지도 API 키 사용
- 지도 위에서 좌표 직접 보정

## 오프라인 처리

오프라인 상태에서도 이미 좌표가 저장된 매장은 지도에 표시할 수 있다.
단, 지도 타일은 브라우저 캐시에 없으면 보이지 않을 수 있다.

오프라인 안정성을 높이려면:

- 현장 출발 전 사무실에서 지역별 지도 보기 실행
- 주소 좌표를 미리 저장
- 필요한 지역의 지도 타일을 미리 열어 캐시에 남겨둠

완전한 오프라인 지도는 별도 지도 타일 저장 구조가 필요하므로 1차 범위에서는 제외하는 것이 좋다.

## 변경 이력 운영

앞으로 기능을 수정할 때는 루트의 `CHANGELOG.md`에 다음 내용을 추가한다.

- 날짜
- 커밋 해시
- 변경된 기능
- 사용자에게 보이는 변화
- 데이터 구조 변경 여부
- 주의사항 또는 되돌릴 기준 커밋

앞으로 커밋할 때는 가능한 한 `CHANGELOG.md`에 해당 커밋의 목적과 변경 내용을 함께 남긴다.
